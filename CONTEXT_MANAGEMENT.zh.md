# 上下文管理规范

> **核心原则：记忆是灵魂，上下文是生命线！**

---

## 🔑 为什么上下文管理如此重要？

```
没有上下文的开发：
用户：我要开发一个 AI Agent
AI：好的，我们开始...
用户：继续
AI：好的，我们开始...
用户：...
```

```
有上下文的开发：
用户：我要开发一个 AI Agent
AI：好的，分析需求，配置环境，开始...
用户：继续
AI：上一次我们完成了环境配置，现在开始架构设计...
用户：👍
```

---

## 📊 标准上下文结构

### 每次回复必须包含：

```markdown
## 📊 当前项目上下文

### 项目基本信息
- 项目名称：[项目名]
- 项目类型：[类型，如 AI Agent、Web App、数据平台]
- 创建时间：[时间]

### 已配置资源
- ✅ 环境：[已配置的工具]
- ✅ 技能：[已加载的技能]
- ✅ 工具：[已配置的服务器]
- ✅ 提示词：[已使用的模板]
- ✅ API：[已接入的服务]

### 当前进度
- [ ] 步骤 1：需求分析
- [ ] 步骤 2：环境配置
- [ ] 步骤 3：架构设计
- [ ] 步骤 4：开发实现
- [ ] 步骤 5：测试验证

### 上一次对话摘要
[简要描述我们上一次做了什么]

---

## 本次执行内容
[你正在做什么]
```

## 🎯 上下文的生命周期

### 阶段 1: 初始化
```
用户：我要开发一个 AI Agent
  ↓
创建上下文：
{
  "project": "AI Agent 项目",
  "type": "AI Agent",
  "created": "2026-05-20",
  "resources": {
    "env": [],
    "skills": [],
    "tools": [],
    "prompts": [],
    "apis": []
  },
  "progress": {
    "analysis": false,
    "env": false,
    "design": false,
    "dev": false,
    "test": false
  },
  "last_action": "用户初始化项目"
}
```

### 阶段 2: 更新
```
每次执行操作后：
1. 更新 resources 字段
2. 更新 progress 字段
3. 更新 last_action 字段
4. 输出到回复中
```

### 阶段 3: 回顾
```
每次对话开始前：
1. 读取上次的上下文
2. 理解当前进度
3. 从上次的位置继续
```

## 🚀 上下文使用流程

### 每次回复的标准流程

```
1. 回顾上次的上下文
   ↓
2. 理解当前进度
   ↓
3. 执行本次操作
   ↓
4. 更新上下文
   ↓
5. 输出（包含完整的上下文）
```

---

## ⚠️ 上下文管理的注意事项

### 必须做的 ✅
- ✅ 每次回复都必须包含当前项目上下文
- ✅ 每次更新进度时，打勾已完成的步骤
- ✅ 每次配置新资源后，更新资源列表
- ✅ 简要记录上一次做了什么
- ✅ 明确说明本次正在做什么

### 绝不能做的 ❌
- ❌ 不要忘记包含上下文
- ❌ 不要丢失进度信息
- ❌ 不要让用户重复说明
- ❌ 不要从第一步重新开始（除非用户要求）
- ❌ 不要假设用户记得上次做了什么

## 🔧 上下文压缩技巧

当对话很长时，可以适当压缩，但核心信息必须保留：

```markdown
## 📊 当前项目上下文

### 项目：[项目名] ([类型])
### 进度：[当前步骤]
### 资源：[简要列表]
### 上次：[简要摘要]
```

但**第一次**和**关键节点**必须显示完整上下文！

## 🚨 强制上下文规则

### 上下文丢失时的处理

```
用户：继续
AI：（看不到上下文）

必须回复：
## 📊 当前项目上下文

### [???] | [???] | [进度：?/5]
- 上次：未找到上下文，需要澄清
- 下一步：请求恢复上下文

---

⚠️ 我注意到上下文丢失了。为了正确继续，请提醒我：

1. 项目名称是什么？
2. 是什么类型的项目？
3. 我们上一步做到哪里了？

或者说"新项目"重新开始。
```
